Midpoint                                                

(Q1) What is integer overflow?

Integer overflow happens when the value of a number is too big to be represented in the bits allocated to store that number.  During the execution of a program, integer overflow can occur when the result of an arithmetic operation, such as multiplication or addition, exceeds the maximum size of the integer type used to store it. An integer overflow causes "undefined behaviour", meaning that compilers conforming to the standard may do anything they like from completely ignoring the overflow to aborting the program.  Most compilers ignore the overflow, resulting in an unexpected result being stored.

(Q2) How do you detect it? (In C and Java 1.7 this is a trick question.)

In the table below, there is an outline of five programming languages the deal with integer overflows in different manners.  

Programming Language

How the deals with Integer Overflow

C

In C, the processor will only use the bits that it has, only taking the true result and throwing out any bits that do not fit. This causes the program to give unexpected output.

Java

In Java, much like C, integer overflows are not detected/ignored by compilers, resulting in an unexpected or erroneous result being stored.

Python

Python will detect that the result will not fit and will automatically allocate space for a larger number.

Ruby

Ruby, in the same manner as Python will notice that the result will not fit and will automatically allocate space for larger numbers.

Swift

Swift will throw an error if an integer overflow is detected, unless you have explicitly indicated that overflowing integers should be truncated to fit.

Integer overflows cannot be detected after they have happened, so there is no way for an application to tell if a result it has calculated previously is in fact correct.  This can get dangerous if the calculation has to do with the size of a buffer or how far into an array to index. Of course most integer overflows are not exploitable because memory is not being directly overwritten, but sometimes they can lead to other classes of bugs, frequently buffer overflows.

(Q3) How do you prevent it?

Listed below are three main techniques that are used for detecting unintended integer overflow by using precondition testing, upcasting and BigInteger. In addition to the three main techniques it is important to take advantage of overflow checking features provided by the compiler or interpreter.

  • Precondition testing → The precondition testing technique requires the input to each arithmetic operator to have a different precondition test to ensure that overflow cannot occur. Throw an ArithmeticException when the operation would overflow if it were performed; otherwise, perform the operation. This approach is more time consuming, difficult to implement and harder to monitor compared to the other two approaches.

  • Upcasting →  The upcast technique casts the inputs to the next larger primitive integer type and performs the arithmetic in the larger size. It is then required that each intermediate result is checked for overflow of the original smaller type and throw an ArithmeticException if the range check fails. The range check must be performed after each arithmetic operation. The larger expressions without per-operation bounds checking can overflow the larger type. The final step to this technique is to downcast the final result to the original smaller type before assigning to a variable of the original smaller type. It is substantially more efficient than using BigInteger. Unfortunately, it cannot be applied to operations involving type long, as there is no bigger type to upcast to.

  • BigInteger →  BigInteger is the easiest of the three techniques to perform. The technique requires the input to be converted into objects of type BigInteger and perform all arithmetic using BigInteger methods. The arithmetic operations implemented as methods of this type cannot overflow; instead, they produce the numerically correct result. Operations on objects of type BigInteger can also be significantly less efficient than operations on the original primitive integer type.Consequently, compliant code performs only a single range check just before converting the final result to the original smaller type and throws an ArithmeticException if the final result is outside the range of the original smaller type.

(Q4) Why are you confident your code is correct?

If the two inputs are greater than the minimum legal integer and less than the maximum legal integer, adding them and then halving ((x + y) / 2), could result in integer overflow in the addition step, as the result of x + y could easily be out of the legal range for an int in c. We have avoided this by separating the problem into two cases.

When the two inputs have different signs, the result from the addition step has to be in the legal integer range. Since one input is less than zero and one is greater than zero, the result from adding the two inputs has to lie somewhere between the two inputs. Since the two inputs are somewhere in the acceptable range, the result has to be too. In this case, our code uses the plain old (x + y)/2 strategy to find the midpoint.

If the two integers have the same sign, adding them could cause an overflow. In our code, we halve the inputs before adding the two halves together, and avoid the potential overflow in the addition step. The result from halving each input cannot be outside the legal integer range if each input is a legal integer (x/2 < x). The problem with this strategy is that it causes off by one errors if the inputs are odd numbers. The result from halving an odd number is rounded towards zero, and since halving twice means rounding 0.5 to 0 twice, you lose 1. We solved this by adding 1 back if both inputs are odd and positive.

(Q5) How hard was it to write a bulletproof version of this little function?

The challenge was splitting the problem into possible cases. We started just by halving and adding((x / 2) + (y / 2)), but found this resulted in off by one errors. We then went through all the possible combinations of even/odd and positive/negative to decide when to add one or minus one and when to just use (x / 2) + (y / 2). This got a bit ridiculous because there are 2^4 different combinations (positive and odd, positive and even, negative and odd …). Recognising that adding before halving was possible without resulting in integer overflow when x and y had different signs cut down the number of cases to consider for adjusting for these off by one errors, and after that adding (x % 2 + y % 2) / 2 seemed obvious. So it did end up being quite hard to write such a simple function.

(Q6) Question for reflection: if it’s this hard to get a simple thing like midpoint right, how much confidence can we have in any integer calculation without formal verification or testing? Programming languages like Lisp, Smalltalk, Python, and Perl6 support integers with no fixed size limit; when is it a good idea to use a “safe” language and when is it a good idea to use a numerically “unsafe” language like Java?

As long as we can guarantee that we will always be working within the legal integer range eg. array indices etc., it is a okay to use an “unsafe” language. However, in lots of situations it is hard to guarantee that we will stay within the range of legal integers. It would be better to use a “safe” language when you don’t know how big your input could be, and therefore how big your numbers could get. It is fine to stick to “unsafe” languages and use numeric types other than ints when you know ints won’t be big enough.

32-bit floats are limited in precision. There are infinitely many values between any two values (numbers are continuous), but only so many fractions can be represented by floats (representable floats are discrete). Dividing x y and z by 10, performing arithmetic on x and y, and taking the difference between z and the result returned from hyp(), can all produce potentially inaccurate results via combinations of floating point underflow and roundoff error.

We have run some tests to check which issues affect which iterations. The output from our testing is attached. As soon as x and y are divided by 10, x and y cannot be exactly represented in 32-bit floats, so are rounded up to the nearest storable float (eg. 0.3f = 0.30000001192092895507812500… and 0.4f = 0.40000000596046447753906250...).  All iterations except iteration 0 are affected by this, as these inexact approximations of the inputs are used by hyp().

Z is accurately represented in iterations 0 and 1 (5.0f and 0.5f), but after this, the z value that we use check our hyp function against is also inaccurate.

The second iteration produces no error, even though x and y aren’t accurate representations for 0.3 and 0.4. This is because when the squares of x and y are added, the result is rounded to 0.25, which happens just happens to be correct. Iterations 2-4 produce no error, because the difference in the noise from x and y and the noise in z is truncated to zero, in other iterations (except for the tenth) this is not the case.

In the rest of the iterations, we don’t happen on any precise results for any operations. It becomes hard to separate the noise from each operation, as the noise from each operation is carried through to subsequent operations. In the later iterations, the values of x*x and y*y become very close to the smallest positive single precision float, but never become subnormal.

(Q10) Test these functions with c = 1 and a = 1, 10, . . . , 10^19. One of these functions goes catastrophically wrong; which one and why?

Neither is perfect, but the implementation of heron’s formula is the one that goes catastrophically wrong. The problem here is round-off error. When a gets to 10e+6, the float arithmetic result of 2a + c rounds the correct result (20000000 + 1) down to 2a. This error feeds through, and as a result the final value of s is off by 0.5. This make s = a, which means the result of (s-a)*sqrt(s*(s-c)) will always be zero, since  s - a = 0.
